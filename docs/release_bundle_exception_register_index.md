# Release bundle exception register index surfacing

The `operator_exception_register` diagnostic is now generated by the standard CI diagnostics bundle and should be easy for reviewers to find from the bundle landing page.

## Purpose

The exception register collects blocker, warning, missing-artifact, and review-item signals from generated handoff diagnostics. It is a reviewer aid for analytical handoffs, not an operational prediction, targeting instruction, or certainty claim.

Surfacing it near the top of the release bundle helps maintainers answer three common questions quickly:

1. Is anything blocking handoff, merge, or release?
2. Which warning or review item needs an owner?
3. Which narrow artifact or command should be rerun before another broad workflow attempt?

## Expected index contract

When `release-bundle-index.html` is generated from a diagnostics directory that contains the exception register outputs, reviewers should be able to find:

- `operator-exception-register.md` as the human-readable prioritized queue.
- `operator-exception-register.json` as the machine-readable queue for downstream automation.
- `operator-exception-register.txt` as the copyable terminal or chat summary.

Recommended label language:

- `Prioritized operator exception queue`
- `Machine-readable operator exception queue`
- `Copyable operator exception summary`

Recommended review-order placement:

1. Confirm bundle readiness with `release-health.md`.
2. Use `reviewer-handoff.md` for reviewer context.
3. Review `operator-exception-register.md` for blockers, warnings, and ownership before promoting the next action.
4. Promote `operator-next-steps.md` after known exceptions are understood.
5. Use `triage-summary.md` for CI failures or missing outputs.
6. Verify inventory and hashes with `artifact-manifest.md`.

## Safe-scope reminder

The exception register should remain framed as handoff metadata for lawful defensive, analytical, and reproducibility workflows. It should not be described as live intelligence, an operational truth source, or a guarantee that predictive outputs are correct.

## Implementation guidance

A future code follow-up can update `app/cli/release_bundle_index.py` by adding the three exception-register outputs to `HIGHLIGHTED_ARTIFACTS` and inserting `operator-exception-register.md` into `REVIEW_ORDER_STEPS` before `operator-next-steps.md`.

The matching regression test should ensure the rendered HTML contains the three exception-register links and review-order wording so the artifact does not disappear from the landing page during later maintenance.

## Rollback

This document is additive. Revert this file and its documentation regression test if maintainers choose not to expose the exception register from the landing page.
