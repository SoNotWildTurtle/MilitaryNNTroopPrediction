# Run Decision CI Bundle Artifacts

The standard CI diagnostics bundle now includes the offline next-increment candidate report and the machine-readable run decision record generated by `app.cli.next_increment_candidates`.

## Purpose

These artifacts help recurring maintenance runs and reviewers carry the same decision context forward without relying on memory, chat history, or manually copied notes. They are designed for lawful defensive repository maintenance and reviewer handoff only. They are not operational tasking, targeting guidance, or evidence that any analytical prediction is true.

## Generated files

`scripts/ci_report.sh` writes these files into the configured artifact directory before the release bundle index and manifest are generated:

- `next-increment-candidates.md` - reviewer-friendly Markdown report of candidate increments derived from the local changelog and roadmap.
- `next-increment-candidates.json` - machine-readable candidate matrix with status, novelty score, validation commands, blockers, and safe-scope text.
- `run-decision-record.json` - selected-candidate record with required merge evidence fields, validation plan, merge blockers, compatibility notes, rollback notes, and next follow-up candidate.
- `next-increment-candidates-help.txt` - current CLI help output for reproducibility and reviewer command discovery.

Because these files are generated before `release-bundle-index.html`, `artifact-manifest.json`, and `artifact-provenance-ledger.json`, they are included in the standard local and hosted diagnostics evidence set.

## Local reproduction

```bash
make ci-report ARTIFACT_DIR=ci_artifacts/run-decision-check
```

For a narrower check that does not build the full diagnostics bundle:

```bash
python -m app.cli.next_increment_candidates \
  --markdown-path /tmp/next-increment-candidates.md \
  --json-path /tmp/next-increment-candidates.json \
  --decision-record-path /tmp/run-decision-record.json
```

## Reviewer workflow

1. Open `release-bundle-index.html` first and confirm the next-increment and run-decision artifacts are indexed.
2. Compare `run-decision-record.json` against the pull request body to ensure final head SHA, required hosted checks, local validation, diff review, compatibility, rollback, blockers, and follow-up work were captured before merge.
3. If a hosted check fails, use the validation plan in `run-decision-record.json` as the smallest relevant rerun before broader `make verify` reproduction.
4. If an artifact is missing, rerun `make ci-report` and inspect `artifact-gap-report.md`, `artifact-manifest.md`, and `artifact-provenance-ledger.md` before merging.

## Compatibility and rollback

This change is additive. It adds generated Markdown/JSON/help artifacts to the diagnostics bundle without changing prediction models, API schemas, data ingestion, persisted records, or existing artifact names. Roll back by reverting the CI report wiring and this document; existing standalone `next_increment_candidates` CLI behavior remains compatible.

## Follow-up

A future run can teach the release bundle index or handoff integrity report to surface the run decision record as a first-class review gate instead of relying on generic artifact indexing.
