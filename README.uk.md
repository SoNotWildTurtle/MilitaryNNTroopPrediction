# Визначення пересування військ за допомогою нейронних мереж

[Read in English](README.md)

Цей репозиторій містить базову реалізацію застосунку машинного зору, що прогнозує пересування військ на основі супутникових зображень та даних OSINT.

## Структура
- `app/` – Python-пакет із основними модулями
  - `config.py` – налаштування конфігурації
  - `database.py` – допоміжні функції для MongoDB
  - `data_ingestion.py` – отримання OSINT та супутникових даних
  - `detection/` – обгортки над YOLO
  - `models/` – моделі для прогнозування траєкторій
  - `pipeline/` – сценарії, що поєднують отримання, виявлення та прогнозування
  - `api/` – ендпоінти FastAPI
- `scripts/` – скрипти встановлення та запуску
- `notes.md` – проектні нотатки
- `dev_notes.md` – технічні нотатки
- `goals.md` – дорожня карта
- `MODEL_CARD.md` – опис моделі та сфера застосування
- `OPS_RUNBOOK.md` – процедурні інструкції та обробка тривог

## Використання
Запустіть `scripts/start.sh`, щоб встановити залежності та підняти API:

```bash
bash scripts/start.sh
```

Оскільки під час цього кроку завантажуються пакети, перевіряйте їхні хеші або використовуйте внутрішні дзеркала, щоб уникнути атак ланцюга постачання у разі захоплення інфраструктури.

Скрипт зберігає контрольну суму у `.cache/setup.sha`, тому під час наступних запусків повторно використовується вже встановлене
середовище без повторного завантаження залежностей. Встановіть `FORCE_SETUP=1` (або видаліть файл зі штампом), якщо потрібно
перевстановити пакети після зміни вимог.

Скрипт спочатку встановлює залежності для CPU. Якщо виявлено GPU, після цього автоматично завантажуються пакети з підтримкою CUDA, такі як PyTorch і повний TensorFlow.

Перевірте встановлені залежності та налаштування сповіщень за допомогою CLI перевірки середовища:

```bash
python -m app.cli.system_check
```

Ключ `--json` надає машинний вивід, а `--skip-optional` дозволяє зосередитися лише на обов'язкових залежностях.

Встановіть `UI_LANG` або використайте ключ `--lang` (наприклад, `--lang uk`), щоб перекласти текст панелі для українських операторів під час запуску інтерактивних CLI.

Після запуску сервер доступний за адресою `http://localhost:8000` з такими ендпоінтами:

- `POST /predict/{area}` – запуск виявлення та прогнозу для області
- `GET /detections/{area}?limit=10` – отримання останніх виявлень
- `GET /predictions/{area}?limit=10` – отримання прогнозів траєкторій

Працюючий сервер також надає мінімальний веб-інтерфейс за шляхом `/gui/`. Відкрийте `http://localhost:8000/gui/` у браузері, щоб переглядати виявлення та прогнози без командного рядка.

Додаткові модулі обробляють знімки Sentinel Hub та CLI-сценарії:
- `satellite/` – завантаження зображень Sentinel та конвеєр інференсу
  - `movement_history.py` – запити до MongoDB щодо останніх позицій підрозділів
  - `pipeline/run_real_time_pipeline.py` – CLI для отримання зображень та запуску виявлення
  - `pipeline/realtime.py` – збереження виявлень і прогнозів у MongoDB
    - `detection/ground_troop.py` – покращене виявлення наземних військ на складних зображеннях
    - `drones/live_feed.py` – обробка відео з дронів у реальному часі
    - `detection/lidar_detector.py` – виявлення військ, техніки та дронів за LIDAR‑даними з позначкою укриття
    - `detection/camera_detector.py` – виявлення військ, техніки та дронів з камер
    - `detection/acoustic_detector.py` – перетворює журнали акустичних ознак у виявлення
    - `detection/sensor_fusion.py` – поєднання виявлень з камер і LIDAR; `detect_fused_objects` запускає обидва сенсори для піхоти, техніки та дронів
    - `detection/clip_identifier.py` – нульовий класифікатор зображень на основі CLIP
    - `detection/unified_identifier.py` – єдиний класифікатор для піхоти, техніки та дронів
    - `cli/extend_unified_model.py` – додає нову ціль до об'єднаного класифікатора, розширюючи вихідний шар за міткою оператора
    Результати виявлення містять поле `doctrine` і зберігаються у колекції `detections`. Прогнози траєкторій зберігаються у `predictions`.

Крок злиття впевненостей об’єднує оцінки детектора з класифікаторами військ, дронів і техніки для підвищення довіри до кожного визначення.
Модуль LIDAR забезпечує електромагнітне виявлення, позначає чи об'єкти під укриттям, а модуль злиття сенсорів поєднує результати камер і LIDAR для підвищення впевненості.
Злиття сенсорів тепер застосовує ваги надійності для камер, LIDAR і Bluetooth та повертає рівень невизначеності для кожного класу.

Каталог `data/builtin_space` вже містить кілька космічних сцен. Кожен файл відповідає відомим російським підрозділам, танковим
колонам або російським та іранським дронам, що дозволяє відпрацьовувати процеси без зовнішніх знімків. CLI `space_identification`
показує перелік сцен і дає змогу переглянути детальні ідентифікації для вибраної сцени.

### Каталог об'єктів
Відстежуйте кожне навчальне зображення з його класом та оцінкою впевненості:

```bash
python -m app.cli.item_catalog --add ITEM_ID CLASS SCORE
python -m app.cli.item_catalog --list
```

Перелічіть вбудовані сцени або перегляньте ідентифікації конкретної сцени:

```bash
python -m app.cli.space_identification
python -m app.cli.space_identification ru_troops_kherson
```

Елементи зберігаються у CSV-файлі, щоб оператори могли впорядковувати їх за класом і переглядати впевненість моделі. `pseudo_labeler` автоматично реєструє кожне позначене зображення з його оцінкою.

### Оперативне зведення

Сформуйте консолідований звіт, що поєднує статистику детекцій, оцінку загроз кластерів
та останню активність моделей:

```bash
python -m app.cli.intel_brief --area donetsk --hours 12
```

Скористайтеся `--limit`, щоб обмежити кількість записів у кожному розділі, а
`--raw` поверне JSON замість форматованих таблиць. Некоректні значення (нуль або
від'ємні числа) зупиняють виконання з підказкою. Якщо область не вказана, звіт
агрегує детекції та прогнози на всіх напрямках.

Той самий звіт доступний і через API для панелей або зовнішніх систем:

```http
GET /intel/brief?area=donetsk&hours=12&limit=25
```

FastAPI валідуює параметри, повертаючи `400`, якщо запит виходить за допустимі
межі.

Окрім простих підрахунків, звіт тепер аналізує **якість детекцій**. Він
розраховує зважену середню впевненість активних класів, підсвічує мітки з
рівнем нижче 0.55 та виявляє класи, що з'являються зрідка або без телеметрії
впевненості. Такі класи автоматично потрапляють до рекомендацій, щоб команда
вчасно запланувала калібрування сенсорів і збалансувала збір даних до появи
«сліпих зон». Метрики якості детекцій відображаються в таблицях CLI та в блоці
`insights`, тож дашборди можуть відстежувати дрейф впевненості з часом.

Звіт тепер містить розділ **оперативного темпу**, що стисло показує кількість
подій, швидкість детекцій на годину та частку подій з прогнозами. Якщо система
спостерігає сплеск детекцій або менше половини подій отримали прогноз, CLI та
API автоматично додають відповідні рекомендації, щоб аналітики швидко
відреагували на навантаження чи збої інференсу.

Додатково з'явився блок **свіжості даних** для детекцій, прогнозів і кластерів.
Кожне джерело показує останній відомий timestamp, вік даних у хвилинах та
статус (*актуальні*, *попередження* чи *застарілі*). Найстаріший потік виділяється
окремо, щоб оператори одразу бачили, де виник збій надходження, а для застарілих
джерел автоматично формується рекомендація щодо відновлення пайплайна.

Зведення також містить **оцінку стану**, що поєднує оперативний темп, рівні
загроз, свіжість даних і точність фідбеку в єдиний індикатор ризику. Аналітики
отримують короткий підсумок поточного рівня ризику, рівня довіри до даних та
ключових чинників, які впливають на показники. Якщо ризик піднімається до
*високого* чи *критичного* рівня, інструмент автоматично радить активувати
план реагування, а низька довіра підсвічує задачі з відновлення телеметрії.

На додачу до оцінки стану, звіт формує **оперативну позу**, щоб чергові одразу
розуміли, у якому режимі працювати наступну зміну. Поза об'єднує рівень ризику,
темп, свіжість телеметрії та кластерні загрози й підказує, чи потрібно просто
*моніторити*, *посилювати*, *стабілізувати* чи *відновлювати* роботу. Кожен
підсумок містить фокус на найближчий плановий горизонт, рівень довіри з оцінки
стану та ключові чинники, що спричинили рекомендацію, допомагаючи синхронізувати
роботу штабу й аналітичної групи.

Щоб перетворити ці висновки на плани зміни, додано блок **готовності до
реагування**. Він поєднує позу, ризик, свіжість телеметрії, точність фідбеку та
навантаження кластерами, щоб порекомендувати, скільки людей поставити на чергування,
скільки годин тримати посилений режим і які дії виконати першочергово
(відновити телеметрію, провести калібрування аналітиків, розгорнути групи
швидкого реагування). CLI показує готовність у таблиці з драйверами та пріоритетами,
а API повертає ті самі підказки для систем автоматичного масштабування змін.

Аналітичне навантаження теж може стати вузьким місцем, тому у звіті з'явився
блок **навантаження на аналітиків**. Він порівнює останні детекції та прогнози,
враховує середню впевненість, точність фідбеку та поточний рівень готовності і
показує, чи черга прогнозів збалансована, зростає чи вже перейшла в критичний
беклог. Блок підраховує кількість прогнозів у черзі, незіставлені детекції та
орієнтовний час ліквідації черги, щоб керівники змін розуміли, коли потрібно
посилювати склад. За пікового навантаження звіт публікує драйвери, формує
цільові кроки (розібрати чергу, перевчити прогнози, провести калібрування
фідбеку) та додає компактне `insight`, щоб панелі могли відображати цей тиск
поруч із позою та готовністю.

Оскільки для відновлення зазвичай потрібна взаємодія кількох команд, звіт тепер
формує чергу **пріоритетів підтримки**. Вона поєднує сигнали готовності, постави,
навантаження, свіжості даних, якості детекцій і таблицю прогалин, щоб позначити
підрозділи (телеметрія, модельні групи, сенсорні інженери, офіцери зв'язку,
команди підтримки аналітиків), яким слід втрутитися, і визначити терміновість.
Кожен запис фіксує причину, за потреби — часовий горизонт, та конкретні кроки,
тож чергові можуть одразу сконтактувати потрібних фахівців. Підсумок відображається
у полі `insights` та у CLI, забезпечуючи швидку ситуаційну обізнаність під час
передачі зміни.

Не менш важливо розуміти, наскільки телеметрії можна довіряти. Звіт тепер
публікує **індекс довіри до розвідданих**, що поєднує точність фідбеку, якість
детекцій, свіжість потоків, відкриті прогалини та загальну оцінку здоров'я в один
бал. Блок показує рівень довіри, числовий результат і ключові компоненти
(зважену впевненість, частку активних класів, перелік застарілих або проблемних
потоків, кількість критичних і значних прогалин), щоб аналітики бачили першопричину
просідання. Драйвери та рекомендовані дії додаються до загального списку
рекомендацій і відображаються у CLI, допомагаючи розставити пріоритети між
відновленням телеметрії, калібруванням чи координацією інциденту до того, як
постраждає якість рішень.

Щоб заздалегідь планувати наступні зміни, зведення тепер формує блок
**оперативного прогнозу**. Він синтезує дані про поставу, готовність,
аналітичне навантаження, свіжість телеметрії, рівень впевненості та загрози у
єдину оцінку з горизонтом планування та фокусними напрямами. Аналітики одразу
бачать, чи перебуває середовище у стані чергування, стабілізації, посиленого
спостереження, швидкого реагування або майже ескалації, і що саме до цього
призводить. CLI відображає блок із фокусами, драйверами та діями, а API
публікує підсумок у `insights.operational_outlook`, тож дашборди можуть
відстежувати прогноз поруч із темпом, готовністю та прогалинами.

Щоб перейти від оцінок до конкретних наказів, зведення формує
**чергу командних директив**. Синтезатор поєднує оперативний прогноз, поставу,
готовність, навантаження, пріоритети підтримки, індекс довіри, оцінку здоров’я
та прогалини, ранжуючи кроки як невідкладні, на наступну зміну або для
моніторингу. Блок публікує зведений статус (`monitor`, `focus`, `accelerate`,
`escalate`), найкоротше планувальне вікно, кількість директив і команди для
координації, щоб штаб одразу бачив, кого мобілізувати. CLI показує таблицю з
джерелами, контекстом і вікнами підтримки, а API повертає детальний блок
`command_directives` і короткий підсумок у `insights.command_directives` для
дашбордів та щоденних брифінгів.

Щоб командири, аналітики та групи підтримки узгоджували ритм сповіщень, зведення
тепер генерує **план комунікацій**. Планувальник аналізує директиви, прогноз,
поставу, готовність, навантаження аналітиків, свіжість телеметрії, рівень довіри,
серйозність прогалин і загрози, щоб визначати періодичність оновлень для кожної
аудиторії, підсвічувати ключові меседжі та пропонувати наступні кроки для
комунікації. CLI показує таблицю аудиторій із каденцією, драйвери та спеціальні
дії, а API повертає повний блок у `communication_plan` і короткий статус із
кількістю аудиторій та каденцією в `insights.communication_plan`, тож дашборди
можуть планувати синхронізації поруч з іншою операційною аналітикою.

Щоб перетворювати підвищені сигнали на чіткі плани дій, зведення також формує
модуль **планування непередбачених ситуацій**. Він поєднує оперативний прогноз,
директиви, поставу, готовність, навантаження аналітиків, пріоритети підтримки,
свіжість телеметрії, якість детекцій, рівень довіри та відкриті прогалини, щоб
описати сценарії, які можуть потребувати запуску. Кожен сценарій містить
тригери, цілі, рекомендованих відповідальних і дії, аби чергові офіцери могли
швидко задіяти потрібні команди. Ключові індикатори й вікна активації відображає
CLI, а в `insights.contingency_plans` доступний стислий виклад, завдяки чому
дашборди показують, наскільки близько організація до виконання ескалаційних
планів.

Щоб ці плани мали ресурсну підтримку, до зведення додано **план забезпечення
ресурсами**. Синтезатор аналізує готовність, навантаження аналітиків, пріоритети
підтримки, свіжість телеметрії, відкриті прогалини, управлінські директиви,
готовність сценаріїв та каденцію комунікацій, аби показати, які команди потребують
персоналу, інженерної чи логістичної допомоги. Він розраховує мінімальне вікно
дозавантаження, агрегує список потреб і формує таблицю розподілу з пріоритетами,
кількістю та вікнами. Rich-CLI показує цей розділ із драйверами та рекомендованими
діями, а API повертає деталі в `resource_sustainment` і стислий підсумок у
`insights.resource_sustainment`, щоб інструменти планування могли заздалегідь
підготувати персонал до пікових або мобілізаційних подій.

Щоб командування отримувало єдину картину загроз, зведення формує
**операційний реєстр ризиків**. Він агрегує найсерйозніші сигнали з готовності,
навантаження аналітиків, координації підтримки, свіжості телеметрії, довіри до
даних, оцінки здоров’я, оперативного прогнозу, директив, планів непередбачених
ситуацій, забезпечення ресурсами, комунікацій і прогалин, щоб порахувати
загальний бал ризику, виділити фокусні напрямки та показати найближче вікно
перегляду. Кожен ризик містить категорію, рівень серйозності, статус, драйвери
та, за потреби, рекомендовану дію, тож штаб може одразу призначити відповідальних.
CLI відображає таблицю ризиків і список драйверів, а API повертає деталі в
`operational_risks` і стислий сніпет у `insights.operational_risks` для
дашбордів і брифінгів під час зміни.

Коли директиви, підтримка або забезпечення починають розходитись, нова панель
**узгодженості командування** допомагає повернути їх в один ритм. Вона зводить
чергу директив, каденцію комунікацій, стан забезпечення, реєстр ризиків,
готовність, навантаження аналітиків, пріоритети підтримки, оперативний прогноз,
позицію та плани непередбачених ситуацій, аби розрахувати бал узгодженості,
показати ключові розриви, спільні фокусні теми та запропонувати найближче вікно
для крос-командного синку. Драйвери узгодженості й агреговані дії допомагають
штабу швидко побачити вузькі місця, що стримують виконання. Rich-CLI показує
узгодженість із прогалинами, драйверами та діями, а API повертає дані в
`command_alignment` і стислий підсумок у `insights.command_alignment`, щоб
дашборди могли координувати зустрічі керівництва.

Щоб надати керівництву єдину контрольну точку, зведення тепер публікує
**індекс забезпечення місії**. Він поєднує готовність, узгодженість,
забезпечення, ризики, антикризові сценарії, комунікації, директиви, оперативний
прогноз, поставу, навантаження аналітиків, пріоритети підтримки, довіру до
розвідданих, загальний стан, свіжість телеметрії та відкриті прогалини в один
бал і статус. У блоці фіксуються чинники, що знижують забезпечення, ключові
фокуси, залежності (вікна підтримки, поповнення, синхронізації) та узгоджені
дії, які потрібно виконати для стабілізації місії. Rich-CLI показує новий розділ
із блокерами, драйверами, сферами уваги та таблицею залежностей, а API повертає
структуру в `mission_assurance` і короткий інсайт, щоб дашборди могли відслідковувати
рівень забезпечення місії поряд із трендами постави, готовності та узгодженості.

Поверх цього додано **пульс оперативної стійкості**, який оцінює гарантію,
готовність, забезпечення, ризики, плани непередбачених ситуацій, комунікації,
узгодженість, навантаження аналітиків, пріоритети підтримки, свіжість
телеметрії, довіру до даних, серйозність прогалин та оперативний прогноз, щоб
зрозуміти, наскільки команди витримають подальші збої. У відповідь формується
перелік підсилювальних факторів, слабких місць, спільних драйверів, вікон
стабільності та дедуплікованих дій, аби штаби розуміли, які важелі зміцнюють
стійкість, а які виснажують ресурс. У Rich-CLI з’являється окрема таблиця зі
статусом, балом і списками підсвічених пунктів, а API повертає структурований
блок `operational_resilience` та короткий інсайт для дашбордів і брифінгів під
час зміни.

Щоб спрогнозувати, чи вистачить цієї стійкості на наступний плановий горизонт,
зведення доповнене синтезатором **операційної безперервності**. Він поєднує
забезпечення, стійкість, логістику, ризики, антикризові сценарії, директиви,
узгодженість, підтримку, готовність, навантаження аналітиків, свіжість
телеметрії, довіру до даних та оперативний прогноз у бал безперервності зі
статусом, переліком обмежень, таблицею ризиків, стабілізуючими факторами,
оцінкою горизонту та оновленими діями. Rich-CLI показує окрему секцію з
таблицями обмежень і ризиків та списком елементів для моніторингу, а API
повертає дані в `operational_continuity` і компактний запис у
`insights.operational_continuity`, щоб дашборди бачили, наскільки команда
наблизилася до критичних вузьких місць.

Щоб керівництво встигало підхопити кермо, коли ці вузькі місця захитались,
зведення тепер обчислює **матрицю готовності до ескалації**. Вона поєднує
командні директиви, стан безперервності, стійкість, гарантію місії, готовність,
навантаження аналітиків, пріоритети підтримки, антикризові плани, комунікаційні
каденції, забезпечення ресурсами, операційний реєстр ризиків, прогалини в
розвідданих, свіжість телеметрії, довіру до даних, оперативний прогноз та
узгодженість штабу, щоб отримати інтегральний бал ескалаційної готовності.
Пейлоад формує чергу ескалаційних шляхів із пріоритетами та тригерами,
розділяє дестабілізуючі сигнали й стабілізуючі фактори, збирає спільні драйвери
та елементи для моніторингу і фіксує найближче вікно перегляду разом із
дедуплікованим переліком дій. У Rich-CLI з’являється виділена секція з таблицею
шляхів, сигналів та рекомендацій, щоб чергові офіцери могли за кілька хвилин
підготувати керівництво, а API повертає матрицю в `escalation_readiness` та
короткий інсайт `insights.escalation_readiness`, що дозволяє дашбордам відстежувати
момент, коли слід скликати керівний осередок для ухвалення рішень.

Щоб замкнути цикл між ескалацією та поверненням до штатного режиму, звіт тепер
будує **дорожню карту оперативного відновлення**. Вона поєднує показники
безперервності, стійкості, гарантії виконання місії, забезпечення ресурсами,
узгодженості штабу, директив, навантаження аналітиків, черги підтримки,
ескалаційний стан, реєстр ризиків, плани на випадок НП, комунікації, рівень
довіри й свіжість телеметрії у зважений план відновлення. Пейлоад підсвічує
критичні залежності, стабілізаційні дії, робочі треки з відповідальними та
фокусами, єдиний список дій і найкоротше вікно відновлення, щоб команди могли
завчасно планувати ресурси. Rich-CLI відображає панель відновлення з таблицями
залежностей і треків, факторами імпульсу та пунктами спостереження, а API
повертає дані в `operational_recovery` і компактний запис у
`insights.operational_recovery` для дашбордів, що координують міжкомандне
відновлення.

Щоб перенести цей імпульс у довгострокові зміни, конвеєр брифінгу додає
**порядок денний операційної трансформації**. Він змішує показники відновлення,
обмеження безперервності, оцінки стійкості та гарантій, плани забезпечення
ресурсами, узгодження командування, черги директив, мобілізації підтримки,
покриття готовності, навантаження аналітиків, оперативний прогноз, реєстр
ризиків, ритм комунікацій, сценарії непередбачених ситуацій, довіру до телеметрії
та загальний стан, щоб оцінити прогрес трансформації. Пейлоад публікує стадію
зрілості, треки фокусу з власниками та діями, швидкі перемоги та довгострокові
ініціативи, підсилювальні чинники, обмеження, індикатори спостереження, метрики
для моніторингу й дедупліковану чергу дій разом із наступним вікном перегляду.
Rich-CLI показує порядок денний через підсумкові таблиці, переліки треків та
списки акцентів, а API повертає структуру в `operational_transformation` і
дублює інсайт у `insights.operational_transformation`, щоб дашборди бачили, як
відновлення переростає в стійкі зміни.

Щоб штаб не втрачав контроль над усіма цими напрямами, у зведенні з’явився
**реєстр оперативного управління**. Він об’єднує імпульс трансформації,
прогрес відновлення, обмеження безперервності, поточну стійкість, гарантію
місії, потреби забезпечення, узгодженість командування, фокус директив,
каденцію комунікацій, серйозність ризиків, черги підтримки, телеметрію
готовності та навантаження, сценарії НП, оперативний прогноз і ескалаційний
постур, щоб оцінити стан управління. Реєстр виносить прогалини у відповідності,
найближчі вікна перегляду, склад наглядових рад, спільні драйвери, фокусні
тематики, індикатори спостереження та дедупліковану чергу дій, тож командири
бачать, яку раду скликати наступною і з якої причини. Rich-CLI показує таблицю
рад зі статусами, темами та наступними кроками, а API повертає структурований
пакет у `operational_governance` і стислий інсайт у
`insights.operational_governance` для дашбордів і передач змін.

Щоб забезпечення не відставало від запитів українських бригад, бриф формує
**профіль підтримки передової**. Він поєднує плани забезпечення, пріоритети
підтримки, телеметрію готовності та навантаження, оцінки безперервності й
стійкості, гарантію місії, перекоси в узгодженні командування, оперативний
прогноз, сигнали реєстру ризиків, попередження про свіжість даних, якість
виявлень і критичні прогалини в об’єкт `frontline_support`. Синтезатор виносить
пріоритетні підрозділи, часові вікна логістичних коридорів, таблиці підтримки
бригад, нотатки для українських операторів і рекомендації з логістики, а
компактний інсайт доступний у `insights.frontline_support`. Rich-CLI показує цей
розділ із таблицями підтримки, сигналами та локалізованими підказками, щоб
українські чергові могли швидко координувати колони та підсилення без переходу
до інших інструментів.

Щоб забезпечити **повну автоматизацію для українських операторів**, бриф тепер
повертає `automation_playbook`. Синтезатор об'єднує командні директиви, ритм
комунікацій, стан управління, здоров'я забезпечення, профіль передової,
готовність, навантаження, пріоритети підтримки, зрілість трансформації,
узгодженість командування, довіру до телеметрії, свіжість даних і прогалини, щоб
визначити режим автоматизації. Він показує тригери, що потребують ручного
підтвердження, формує чергу автоматизованих завдань з відповідальними та
вікнами виконання, нагадує які канали моніторингу тримати відкритими, а також
додає україномовні підказки, щоб кожен автоматизований крок був перевірений
черговим офіцером. CLI відображає таблицю автоматизації, матрицю завдань,
тригери та підказки, а REST-інтерфейс повертає payload у
`automation_playbook` зі стислим інсайтом у `insights.automation_playbook` для
дашбордів, ботів і планувальників змін.

Щоб автоматизація залишалася безпечною, бриф також формує `automation_guardrails`
— перелік поточних обмежень і статус автономності з наступним часом перегляду.
Синтезатор аналізує готовність, навантаження, підтримку передової, управління,
гарантію місії, стійкість, безперервність, відновлення, трансформацію,
забезпечення, підтримку, узгодженість командування, директиви, комунікації,
оперативний прогноз, довіру до даних, якість виявлень, свіжість каналів,
прогалини розвідки, оцінку ризиків і точність фідбеку разом з самим
`automation_playbook`. На основі цих сигналів модуль посилює або послаблює
обмеження, пропонує ручні підтвердження, додає чеклісти українською для чергових
офіцерів та наголошує, які канали моніторингу мають бути активними для
безнаглядових запусків. CLI показує таблицю гвардреєлу, перелік перевірок
безпеки, потрібні оверрайди та кандидати на автоматизацію, щоб українські
оператори швидко оцінювали безпечність перед делегуванням повторюваних задач.
Щоб звести всі сигнали автоматизації в єдиному пункті керування, бриф тепер
повертає payload `automation_mission_control`. Він об'єднує playbook, guardrails,
готовність, навантаження, підтримку передової, ритм рад управління, бойові
директиви, канали комунікації, стан узгодженості, тиск на забезпечення, довіру
до телеметрії, точність фідбеку та часові рамки ескалацій у місійний рейтинг з
рівнями нагляду. Модуль підсвічує критичні бар'єри, канали місійного зв'язку,
вимоги до передачі зміни, дії для наставників і набір українських підказок, щоб
чергові змогли підготувати наступну зміну за кілька секунд. Дашборди читають
стислий статус у `insights.automation_mission_control`, а Rich-CLI показує
таблицю місійного контролю, фокусні напрями, watch-елементи й локалізовані
підказки, щоб українські оператори залишалися синхронізованими під час повністю
автоматизованих запусків.

Щоб зафіксувати, коли автономні дії можна залишати без нагляду, звіт формує
payload `automation_autonomy`. Він поєднує playbook, guardrails, місійний
контроль, готовність, навантаження, підтримку переднього краю, управління,
узгодженість, директиви, комунікації, стійкість, безперервність, відновлення,
гарантію, довіру до телеметрії, свіжість каналів, ризики й розвідпрогалини в
єдину оцінку автономності з чіткими вимогами до нагляду. Синтезатор розділяє
довірені та обмежені задачі, показує драйвери посилення й ризики, обчислює
найближче вікно автономії та збирає канали моніторингу й резервні протоколи,
які мають бути під рукою у українських операторів. Також публікуються
локалізовані заходи безпеки та підказки для тренувань, щоб фіксувати кожен
оверрайд і попереджати чергового офіцера перед розширенням автономних вікон.
Rich-CLI виводить таблицю автономності, розподіл задач, список спостереження та
україномовні інструкції, а панелі отримують структурований блок через
`insights.automation_autonomy` разом із дедуплікованими рекомендаціями.

Щоб закріпити безпечні аварійні сценарії, бриф тепер видає payload
`automation_failsafes`. Він звіряє playbook, guardrails, mission control,
autonomy, готовність, навантаження, підтримку передової, забезпечення,
пріоритети підтримки, ритм управління, гарантію місії, стійкість,
безперервність, відновлення, трансформацію, командні директиви, комунікації,
таймінги ескалацій, довіру до телеметрії, якість виявлень, свіжість каналів,

### SMS та email-сповіщення

Щоб увімкнути SMS-сповіщення з панелі, додайте облікові дані Twilio до середовища або `.env`:

```
TWILIO_ACCOUNT_SID=your-account-id
TWILIO_AUTH_TOKEN=your-auth-token
TWILIO_FROM_NUMBER=+15551234567
```

Щоб дублювати сповіщення електронною поштою, вкажіть SMTP-реле:

```
EMAIL_SMTP_HOST=smtp.your-domain.local
EMAIL_SMTP_PORT=587
EMAIL_SMTP_USERNAME=alert-bot
EMAIL_SMTP_PASSWORD=super-secret
EMAIL_FROM_ADDRESS=ops-room@your-domain.local
EMAIL_USE_TLS=1  # встановіть 0, якщо реле вже працює через TLS
```

Після налаштування відкрийте на панелі блок **Автоматизація сповіщень**. Там можна:

- обрати класи для моніторингу (переміщення військ, колони техніки, дрони) та за потреби вказати район;
- встановити мінімальну впевненість для спрацювання правила;
- додати списки номерів і адрес (через кому) для SMS та email;
- увімкнути або вимкнути правило, надіслати тестове сповіщення чи видалити його безпосередньо з інтерфейсу.

Правила зберігаються у `data/alert_rules.json`, а API `/alerts`, `/alerts/{id}` та `/alerts/{id}/test` дає змогу автоматизувати їх створення чи аудит.

### Розширення об'єднаного класифікатора
Збільшуйте модель, коли потрібно відстежувати новий клас:

```bash
python -m app.cli.extend_unified_model
```

CLI запитає шлях до збереженої моделі та нову мітку. Додається випадково ініціалізований нейрон, після чого модель можна донавчити на нових зображеннях.

## Середовище
Конвеєр може отримувати зображення з Sentinel Hub. Перед запуском встановіть такі змінні середовища:

```
export SENTINEL_CLIENT_ID="your-client-id"
export SENTINEL_CLIENT_SECRET="your-secret"
export SENTINEL_INSTANCE_ID="your-instance-id"
export OPENAI_API_KEY="your-openai-key"
export SOURCE_CATALOG="sources.json"
export RESOLUTION_SCALE="1.0"  # >1 для детальніших ознак та хмар точок
export FEATURE_RICHNESS="1.0"  # >1 для додаткових бінів і детальнішого HOG
export HIGH_MEMORY_MODE="0"   # встановіть 1, щоб увімкнути максимально багаті ознаки
```

`RESOLUTION_SCALE` дозволяє системам з більшою пам'яттю створювати
деталізованіші хмари точок та HOG-ознаки шляхом збільшення зображень перед
аналізом. `FEATURE_RICHNESS` додатково збільшує кількість бінів гістограми,
деталізацію HOG і густину хмари точок при значенні більше `1`. Якщо ввімкнути
`HIGH_MEMORY_MODE` (або задати більший множник багатства), система рахує
додаткові багаторівневі гістограми, багатомасштабні HOG-ознаки, LBP-текстури,
Lab-гістограми, відгуки ґаборів, низькочастотні коефіцієнти Фур'є, текстурні
метрики GLCM, гістограми з додаткових проекцій, Hu-моменти, коваріації каналів,
енергії вейвлетів, сіткові Lab-статистики, вбудовування EfficientNet та
субпіксельні/підсилені краями хмари точок для максимальної деталізації.

Якщо змінні не задані, будуть використані тестові зображення.

Файл `.env.example` містить список необхідних змінних; скопіюйте його в `.env` та налаштуйте значення для свого середовища.

Щоб запустити автономний конвеєр з командного рядка:

```bash
python -m app.pipeline.run_real_time_pipeline AREA path/to/model
```
